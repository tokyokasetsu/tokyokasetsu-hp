# 要件定義・設計思想

## 要件定義ワークフロー

### 録音→要件定義
- クライアントとの会話を録音 → 文字起こし → Claude で要件定義書に整形
- ヒアリング漏れを防ぎ、認識齟齬を最小化

### 要求定義 vs 要件定義

| | 要求定義 | 要件定義 |
|---|---|---|
| 誰が | クライアント・ユーザー | 開発チーム |
| 内容 | 「こうしたい」（願望・目的） | 「こう作る」（仕様・制約） |
| 場面 | チャットでの対話がまさにこれ | 要求を受けて技術的に整理 |
| フェーズ | 発散（やりたいことを広げる） | 集約（技術的に絞り込む） |

**このチャット（Claude Code との対話）は要求定義の場**。ユーザーの「こうしたい」を受け取り、技術的な要件に変換するのが Claude の役割。

### 非機能要件6分類（IPA準拠）

定量的に記述すること:

| 分類 | 内容 | 例 |
|---|---|---|
| 可用性 | 稼働率・障害復旧時間 | 稼働率99.9%、RTO 4時間 |
| 性能・拡張性 | 応答時間・同時接続数 | レスポンス2秒以内、同時100接続 |
| 運用・保守性 | 監視・バックアップ・更新 | 日次バックアップ、月次パッチ |
| 移行性 | データ移行・環境移行 | 旧DB→新DB移行、ダウンタイム1時間以内 |
| セキュリティ | 認証・暗号化・アクセス制御 | TLS1.3、二要素認証必須 |
| システム環境 | インフラ・OS・ミドルウェア | AWS東京リージョン、Node.js 20 |

## 設計思想

### 言葉の背景・意図を掴む
- ユーザーの発言を字義通りに受け取らない
- 背景にある目的・課題・感情を読み取って対応する

### クソアプリの原因はセクショナリズム
- 部署間の壁・縄張り意識が品質を下げる
- 全員で「いい作品を作る」意識を共有することが重要

### 認知負荷を下げる
- 返答は簡潔に。情報過多は理解を妨げる
- 必要十分な情報量で応答する
