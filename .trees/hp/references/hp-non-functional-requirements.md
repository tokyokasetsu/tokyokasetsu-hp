| フェーズ | サービス | 用途 |

## 非機能要件 6分類

### 1. 可用性

| 項目 | 定義 | 数値目標 |
|---|---|---|
| **稼働率** | 年間を通じてHPが閲覧できる割合 | **99.5%以上**（さくらインターネット標準SLA準拠） |
| **RTO（目標復旧時間）** | 障害発生から復旧完了までの時間 | **24時間以内** |
| **RPO（目標復旧時点）** | 障害発生時にどの時点のデータまで復旧するか | **前日バックアップ時点**（最大24時間分のデータ損失を許容） |
| **メンテナンス停止** | 計画停止の通知リードタイム | **24時間前に仁さんへ報告してから実施** |

> **補足（仁さん向け）**:  
> 稼働率99.5%は年間約44時間まで停止を許容する水準です。さくらインターネットのスタンダードプランはこの水準を満たしています。  
> RTOの24時間は「障害を知ってから1日以内に直す」という目標です。

---

### 2. 性能・拡張性

| 項目 | 定義 | 数値目標 |
|---|---|---|
| **ページ表示速度** | トップページの初回表示完了時間（FCP） | **3秒以内**（4G回線・モバイル環境） |
| **想定月間PV** | 月間ページビュー数 | **1,000〜10,000PV** |
| **同時接続数** | 同時にアクセスしても問題ない接続数 | **50接続同時**（さくらインターネット スタンダードプランで対応可能） |
| **画像ファイルサイズ上限** | 1枚あたりの最大サイズ | **300KB以下**（WebP形式推奨） |
| **拡張性** | PVが10,000を超えた時の対応方針 | **Vercel or CloudFront CDN導入を検討**（監視mdに報告して仁さんに確認を取る） |

> **補足（仁さん向け）**:  
> 月間1,000〜10,000PVはコーポレートサイトとして一般的な規模感です。さくらインターネット スタンダードプラン（月額524円）で十分に対応できます。  
> FCP（First Contentful Paint）はGoogleが定義する「ページが最初に何かを表示するまでの時間」です。3秒以内が合格水準です。

---

### 3. 運用・保守性

| 項目 | 定義 | 数値目標・ルール |
|---|---|---|
| **バックアップ頻度** | HTMLファイル・画像・設定ファイルのバックアップ | **GitHubへの毎push時に自動バックアップ**（backup-checkerが確認） |
| **パッチ・更新** | ライブラリ・依存パッケージの更新 | **月1回確認**（セキュリティパッチは発見後48時間以内に適用） |
| **ページ死活監視** | 全公開URLが正常に表示されているか | **deploy-checkerエージェントが定期確認** |
| **コンテンツ更新手順** | HPの文章・画像を更新する手順 | **このチャットで仁さんが指示→私が修正→GitHub push→自動反映** |
| **障害検知** | 表示崩れ・リンク切れ・サーバーエラーの検知 | **deploy-checkerが検知次第、仁さんに即時報告** |

> **補足（仁さん向け）**:  
> GitHubへのpushが自動バックアップになります。何かあっても過去のcommit履歴からいつでも復元できます。

---

### 4. 移行性

| 項目 | 定義 | 数値目標・ルール |
|---|---|---|
| **GitHub Pages → さくらインターネット移行** | 開発フェーズから本番フェーズへの切り替え | **ダウンタイム1時間以内**（移行手順書を事前に作成してから実施） |
| **ドメイン切り替え** | 独自ドメインのDNS切り替え時間 | **DNS伝播を考慮して最大72時間の並行稼働期間を設ける** |
| **データ移行** | 静的ファイル（HTML・CSS・JS・画像）の移行 | **GitHubリポジトリからcloneして全ファイルを一括移行** |
| **将来の移行先変更** | さくら→他サービス（Vercel等）への移行 | **静的サイト構成を維持することで移行コストを最小化する** |

> **補足（仁さん向け）**:  
> 移行作業は私が計画書を作ってから実施します（Planモード必須）。仁さんが確認してOKを出した後に実行します。DNSの切り替えは最大3日かかることがありますが、その間はどちらのサーバーでもHPが見られる状態を維持します。

---

### 5. セキュリティ

| 項目 | 定義 | 数値目標・ルール |
|---|---|---|
| **通信暗号化** | HTTPSの使用 | **TLS1.2以上必須**（さくらインターネットの無料SSL「Let's Encrypt」を使用） |
| **APIキー管理** | 認証情報の管理方法 | **コードへの直書き禁止。環境変数で管理**（security-checkerが毎push前にスキャン） |
| **問い合わせフォーム** | スパム・不正送信対策 | **reCAPTCHA v3を実装**（フォーム実装時に必須） |
| **不正アクセス対策** | 管理画面・設定ファイルへのアクセス制限 | **さくらインターネットの.htaccessで.git・設定ファイルへのアクセスをブロック** |
| **脆弱性スキャン** | セキュリティリスクの定期確認 | **リリース前にsecurity-checker（Opusモード）でフルスキャン実施** |
| **個人情報** | 問い合わせフォームから取得する個人情報の扱い | **プライバシーポリシーページを設置。取得項目は最小限（氏名・メールアドレスのみ）** |

> **補足（仁さん向け）**:  
> TLS（HTTPS）はウェブサイトの通信を暗号化する仕組みです。URLが「https://」で始まっていればOKです。さくらインターネットでは無料で設定できます。

---

### 6. システム環境

| 項目 | 定義 | 仕様 |
|---|---|---|
| **ホスティング（開発）** | 開発・確認フェーズ | GitHub Pages（無料） |
| **ホスティング（本番）** | 本番公開フェーズ | さくらインターネット スタンダードプラン |
| **ドメイン** | 本番ドメイン | さくらインターネットで取得・管理（詳細は仁さんと別途決定） |
| **静的ファイル構成** | フロントエンド技術スタック | HTML / CSS / JavaScript（フレームワークなし・静的サイト） |
| **対応ブラウザ** | 動作保証ブラウザ | Chrome最新版・Safari最新版・Edge最新版・Firefox最新版 |
| **対応デバイス** | レスポンシブ対応 | PC（1280px以上）・タブレット（768px〜1279px）・スマートフォン（〜767px） |
| **バージョン管理** | ソースコード管理 | GitHub（共有ハブ）。mainブランチが本番相当。 |
| **CI/CD** | 自動デプロイ | GitHub Pages：GitHub Actions自動デプロイ。さくら：手動またはFTPアップロード。 |

> **補足（仁さん向け）**:  
> 「静的サイト」とはWordPressのようなデータベース不要のシンプルな構成です。表示速度が速く、セキュリティリスクが低く、さくらインターネットのスタンダードプランで十分に動作します。

## 要件変更ルール

- **数値目標の変更は仁さんの承認後のみ実施**（私が変更することは禁止）
- **ホスティングサービスの変更も同様**（金銭に関わる決定のため仁さんの判断が必要）
- 定義漏れを私が発見した場合はこのチャットで仁さんに確認を取る（項目23）
- フェーズ移行時（GitHub Pages→さくら）は移行計画書を作成してから仁さんに報告する
